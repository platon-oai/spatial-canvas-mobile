# Prototype Instructions

Run the local server yourself and open the preview in the in-app browser. Do not give the user server-start instructions when you can run it.

Before making substantial visual changes, use the Product Design plugin's `get-context` skill when the visual source is unclear or no longer matches the current goal. When the user gives durable prototype-specific design feedback, preferences, or decisions, record them in `AGENTS.md`.

When implementing from a selected generated mock, treat that image as the source of truth for layout, component anatomy, density, spacing, color, typography, visible content, and hierarchy.

## Durable interaction decisions

- Two-finger trackpad scrolling pans the spatial canvas; trackpad pinch and Command/Control + wheel zoom around the cursor.
- Trackpad/wheel navigation is intentionally responsive: wheel panning uses a 1.75× delta multiplier and wheel/pinch-event zoom uses `0.0034` sensitivity (2× the original response).
- Pointer-following feedback (pan, drag, resize, marquee, and snap guides) must update through the retained MotionValue/DOM renderer without waiting for React state.
- Dragging and resizing both expose live edge-alignment guides against nearby rendered items; keep those guides in the same retained frame as item geometry.
- Auto-organize is an anchored local tidy, never a global row-major repack. Infer only nearby rows/columns, preserve reading order and rotation, equalize local edge gutters, keep isolated/distant cards at their original neighborhood, and use minimum-axis displacement only for real overlaps.
- A non-overlapping tidy lane has an 80px soft movement budget and splits into smaller local segments before exceeding it. Boards with 750+ targets solve in a module worker so organizing an infinite canvas cannot block pointer/camera rendering.
- Resize-corner indicators keep a screen-space stroke and expanded hit boundary at every canvas zoom. Their arc curvature alone scales with live camera zoom so it continues to match the item's world-space corner radius.
- Resize-corner hit expansion is intentionally modest at 2px per side; do not let the resize decision boundary intrude far into or outside nearby cards.
- Resize-corner curvature scales from each arc's outer offset corner, keeping the perceived card-to-indicator gap invariant while zoom changes.
- Selecting an item beneath an already-hovering mouse/pen reveals its resize-corner indicators immediately; leaving starts a 1-second hide delay. Touch selection must not synthesize hover indicators.
- Ordinary active selection stays at the item's canonical 1× scale. Only two or more items selected by a real drag-marquee receive the retained 1.075× emphasis; every selection still keeps its screen-space white outline, and later click/Shift-click selection returns to 1×.
- Item corner radii remain in world space and scale naturally with the canvas; never inverse-compensate the cards themselves for camera zoom.
- The mobile context toolbar keeps layout centering on a static anchor and runs entrance transforms on an inner motion wrapper; keep it vertically clear of both bottom chrome groups.
- Fullscreen detail actions are portaled into the canvas viewport and fixed to its bottom safe area. Never render that bar inside a transformed item or scrollable document surface.
- The context toolbar anchor spans the viewport and centers with flex layout. Never center a shrink-to-fit animated wrapper with a competing `left: 50%` transform.
- Context-toolbar action-set changes retain semantic button keys, animate the pill's measured size, and move shared actions with layout projection before newly inserted actions resolve. Never mount fixed-width slots into an unanimated intrinsic-width bar.
- Drag snapping is a precision affordance, not continuous resistance: use a 6px capture threshold and 220px nearby search, engage only after a continuous 48ms low-velocity dwell, never newly engage on pointer-up, release immediately during fast travel, and honor Control as a temporary bypass.
- Fullscreen item and folder-shell transitions lay out the destination once, before paint, then animate only compositor-safe x/y, a uniform scale, and clip insets. Never animate width/height every frame or use non-uniform scale; both endpoints must be visually identical to their resting states.
- A settled mobile fullscreen item must merge visually into its matching viewport backdrop: remove the finite card-edge shadow and release paint clipping only after `detailReady`, then restore both immediately before the reverse transition so the action-bar shadow never ends at a visible horizontal seam.
- Board and fullscreen are two states of the same retained `CanvasItemNode` and `SharedItemViewer`. Keep the original preview mounted through open and close; never reintroduce a separate detail-sheet clone or swap the shell component during the transition.
- A document card preview is an exact clipped, uniformly scaled view of the canonical fullscreen document surface. Never use breakpoint/container-query typography that changes line wrapping independently while the shell is moving.
- Authored Notion notes and documents use the entire viewport height in detail, but their reading surface is capped at 680px and centered; on narrower screens it fits inside a stable 24px side gutter. Lay out that capped destination before paint, then animate only x/y, uniform scale, and clip insets from the card.
- Board-to-fullscreen geometry uses a deterministic non-overshooting bounds tween from the retained node's live x/y/width/height. The content DOM must remain mounted and visually continuous for the entire forward and reverse transition; editing controls may activate only after the shell reaches fullscreen.
- Authored editor detail uses a full-height, maximum-680px reader shell centered in screen space. Install that one canonical layout before the flight, keep its inner scale and x-offset constant, and animate only the outer x/y/uniform scale plus clip. Keep `detailReady` mounted through the complete reverse flight and disable editing separately with `detailOpen`; never switch back to preview layout while the shell is moving.
- Rebase a closing shared shell to its canonical board geometry before clearing transition presence. Do not let the final frame swap from a scaled fullscreen layout into a separately laid-out card.
- Web clips use one retained captured-page screenshot surface on the board and in fullscreen. Never swap the screenshot into an unrelated article/editor surface during the morph; capture failures stay in the same retained card with a clear fallback.
- Artifact title/hostname overlays are board-preview chrome only and disappear once fullscreen settles. External-source Open actions belong exclusively in the shared black fullscreen action bar, never as floating buttons over document or web content.
- A web clip title overlay shows one normalized hostname only. Never repeat title/description metadata above it, and allow long hostnames to wrap rather than crop or ellipsize.
- Every canvas item title uses the folder/stack title voice: Inter Display, 15px, weight 500, 1.08 line height, and -0.025em tracking. Keep that shared token across note, document, web, image, folder, and stack cards.
- An asset-backed web clip never re-caches its already-saved PNG. Preload the cached asset, keep the current screenshot painted during handoff, then promote the decoded local PNG without re-entering the loading state.
- Modal motion belongs to the panel's inner transform while centering uses fixed inset/margins or the independent `translate` longhand. Never combine `left: 50%` centering with a Motion-owned `transform`; it pushes half the panel off-screen on phones.
- New modals use the compact Linear-like surface: one clear title, only essential copy and actions, a thin neutral border, modest radius, no glass blur, and a restrained fade/translate transition.
- Route every add, import, rename, and capture prompt through `ModalDialog`; do not create another standalone positioned form or reuse the legacy `url-capture`/`document-import-dialog` layout classes. The shared modal grid owns desktop centering, mobile safe areas, keyboard resizing, and dismissal behavior.
- Keep the retained detail viewer compositor stable across the `detailReady` boundary. Authored editors keep their scroll clipping; imported Office readers deliberately become transparent and unclipped only after settle so their iframe page stream lies directly on the shared fullscreen backdrop.
- Standard Google Workspace embeds are best-effort: keep their iframe inert and covered on the board, enable Storage Access interaction only in detail, and always offer an `Open in Google` fallback. Published `/pub?embedded=true` documents may render directly.
- Imported Office binaries live in the IndexedDB asset table and board items retain only an `assetId`. Lazy-load the DOCX/PPTX/XLSX renderer only when that retained item is visible, so undo snapshots and ordinary canvas startup never clone or download document payloads.
- Imported document pixels are top/left anchored under cover scaling so readable leading edges never crop. Artifact titles and Open actions belong to the retained card viewport, outside any inner media transform. Choose light or dark title chrome from the rendered bottom-region luminance; light artifacts use a light gradient with dark text, while dark artifacts use a dark gradient with white text.
- Imported documents use cover scaling and title chrome only on the board. At settled fullscreen they become flat, zero-radius, full-height readers with no title/description overlay or visible scrollbar. DOCX uses centered white paper pages with real page gaps, XLSX uses a full-viewport grid with a pinned workbook header/formula bar and horizontally scrollable sheet tabs, and PPTX uses a pinned thumbnail rail, one large active slide, and a notes region.
- A same-origin Office iframe is retained from board through detail. Install its fullscreen bounds before the outer FLIP, preserve exact board pixels with one inner preview compensation scale, and crossfade its pre-laid preview/reader layers only after geometry settles. The iframe fills the fullscreen shell and its format-specific reader owns native scrolling; never add a competing host scroll offset.
- Image detail uses one retained crop surface. Its board crop becomes a centered large image through an inner compositor transform while the outer shell FLIPs fullscreen; do not swap the image node or relayout its crop on the first frame.
- Empty editor blocks show the slash-command hint only while that specific block has focus; inactive empty blocks stay visually blank.
- Ordinary editor rows remain transparent in every hover, focus, and menu state. Reserve filled row backgrounds for code blocks.
- Commit canonical React and IndexedDB state once when a gesture ends; reserve Motion springs for discrete transitions such as stacking, opening, and undo/redo.
- Structural undo/redo is atomic across the group shell, child `stackId` values, container `memberIds`, and before/after selection. Pin affected retained nodes for the preparation frame so group creation, folder moves, undo, and redo fan from live geometry and remain interruptible.
- Undo history stores references to immutable item arrays. Never structured-clone the whole board or JSON-stringify item arrays on an interaction path.
- Collapsed stacks and folders use their real retained child cards as visible depth layers; synthetic color-only peek rectangles are not an acceptable substitute.
- Drag a collapsed stack/folder as one retained motion group: the shell alone participates in snapping, its four visible real child layers follow live, and drop commits the shell's final delta to every canonical member exactly once.
- A first folder click selects it; a second click opens its inline contents. Nested item detail must return to its exact in-folder geometry before the folder itself returns to the parent board.
- Folder creation and membership are explicit: Plus creates a new empty folder, while selected leaf items use an Add-to-folder picker with existing destinations, New folder with selection, and Remove from current folder. Never overload the folder icon with rename/remove/create behavior.
- Capture a folder's opening translation and parent camera once. Nested panning must not re-center after camera commits, and closing the folder restores the exact parent camera while the real children collapse.
- Folder-local item coordinates remain canonical when dragging or adding content; never persist the temporary opening translation into item poses.
- Keep viewport culling with overscan enabled so off-screen cards do not add layout, paint, or snapping work.
- Stack and folder transitions retain only destination-page members plus the four visible collapsed layers. Never mount every member just to animate a large group.
- Camera-page culling must use the memoized item lookup and retained stack/folder parents. Never rebuild an all-items map or scan the whole board on each page query.
- Any new wheel, touch, or focus camera motion must be interruptible. New user input adopts the live transform and cancels the prior animation so the camera cannot snap back.
- Only the active fullscreen editor may own selection listeners and editable block DOM. Board previews remain inert representations of the same content.
- Keep image previews responsive (`srcset`/`sizes`) and avoid decoding original multi-megapixel sources for small canvas cards.
- Folder-card preview copy stays hidden while the fullscreen shell is morphing and fades in only after the shell is back at canonical card geometry; do not expose scaled or reflowing preview text during the collapse.
- The canvas itself fills the browser or Electron viewport; do not reintroduce an artificial window frame or fake traffic-light controls.
- On touch devices, one-finger background drag pans and two fingers pan and pinch simultaneously around their live centroid.
- Keep the React/Electron source on `main` in `platon-oai/spatial-canvas-mobile` and publish generated phone-test builds only to its `gh-pages` branch. Run `pnpm publish:pages -- "Update description"`; it builds, replaces only the static artifact on `gh-pages`, and updates the stable `https://platon-oai.github.io/spatial-canvas-mobile/` URL. Do not commit generated `dist` output to `main` or return to manual Sites packaging unless the user explicitly asks.
